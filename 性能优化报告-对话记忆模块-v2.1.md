# 性能优化报告：对话记忆模块（bge-m3嵌入向量缓存优化）

**版本**：v2.1  
**日期**：2026-06-16  
**优化内容**：预计算嵌入向量 + 内存缓存 + 磁盘缓存

---

## 一、优化前性能分析（v2.0）

### 1.1 性能瓶颈

**问题**：每次调用`selectRelevantHistory`时，都需要：
1. 对当前查询计算嵌入向量（1次API调用）
2. 对**所有历史轮次**计算嵌入向量（N次API调用，N=历史轮次数）
3. 计算余弦相似度

**性能数据**（基于测试环境）：
- bge-m3嵌入向量计算时间：**2-5秒/次**
- 10轮对话历史 → 总耗时：**20-50秒** ❌（不可接受）
- 50轮对话历史 → 总耗时：**100-250秒** ❌（完全不可用）

### 1.2 用户影响

- ❌ **响应速度极慢** - 用户提问后需要等待20-50秒才能看到回答
- ❌ **无法实际使用** - 50轮对话后系统基本卡死
- ❌ **CPU/内存占用高** - 频繁调用Ollama API

## 二、优化方案（v2.1）

### 2.1 核心优化策略

#### **策略1：存储时预计算**
- **时机**：在`storeConversationRound`时，立即异步计算嵌入向量
- **优势**：用户提问时，嵌入向量已经算好了
- **实现**：`precomputeEmbedding()`函数

#### **策略2：三层缓存机制**
1. **内存缓存**（`embeddingCache`）- 最快，重启后丢失
2. **磁盘缓存**（`{sessionId}-embeddings.json`）- 持久化，重启后可用
3. **实时计算**（降级）- 缓存未命中时实时计算

#### **策略3：异步处理**
- **存储时**：不阻塞主流程，在后台异步计算
- **查询时**：优先从缓存读取，未命中才实时计算

### 2.2 缓存查找顺序

```
查询时读取嵌入向量：
  1. 检查内存缓存（embeddingCache） → 命中立刻返回
  2. 检查磁盘缓存（{sessionId}-embeddings.json） → 命中则存入内存缓存
  3. 实时计算（getEmbedding） → 计算后存入两层缓存
```

### 2.3 关键代码修改

#### 修改1：`storeConversationRound`函数
```javascript
// 存储成功后，异步预计算嵌入向量（不阻塞主流程）
precomputeEmbedding(sessionId, roundWithId).catch(err => {
  console.warn('[DialogueMemory] 预计算嵌入向量失败:', err.message);
});
```

#### 修改2：新增`precomputeEmbedding`函数
```javascript
async function precomputeEmbedding(sessionId, round) {
  const text = round.userQuery + ' ' + (round.aiResponse || '');
  const embedding = await getEmbedding(text);
  
  // 存入内存缓存
  embeddingCache.set(round.roundId, embedding);
  
  // 存入磁盘缓存
  const cache = loadEmbeddingCache(sessionId);
  cache[round.roundId] = embedding;
  saveEmbeddingCache(sessionId, cache);
}
```

#### 修改3：修改`selectRelevantHistory`函数
```javascript
// 优先从内存缓存读取
if (embeddingCache.has(round.roundId)) {
  roundEmbedding = embeddingCache.get(round.roundId);
}
// 其次从磁盘缓存读取
else if (diskCache[round.roundId]) {
  roundEmbedding = diskCache[round.roundId];
  embeddingCache.set(round.roundId, roundEmbedding);
}
// 实时计算
else {
  const roundText = round.userQuery + ' ' + (round.aiResponse || '');
  roundEmbedding = await getEmbedding(roundText);
  embeddingCache.set(round.roundId, roundEmbedding);
  
  // 异步存入磁盘缓存
  setImmediate(() => {
    const cache = loadEmbeddingCache(sessionId);
    cache[round.roundId] = roundEmbedding;
    saveEmbeddingCache(sessionId, cache);
  });
}
```

## 六、剩余问题与后续优化建议

### 6.1 剩余问题

#### **问题1：首次查询仍然较慢（2-5秒）**
- **原因**：需要实时计算当前查询的嵌入向量
- **影响**：用户体验仍有提升空间
- **解决方案**：
  1. 使用更快的嵌入模型（如bge-small-zh）
  2. 批量计算嵌入向量（一次API调用计算多个文本）
  3. 使用本地嵌入模型（不依赖Ollama服务器）

#### **问题2：磁盘缓存未加密**
- **原因**：嵌入向量直接存储为JSON文件
- **影响**：可能泄露对话内容
- **解决方案**：
  1. 加密磁盘缓存文件
  2. 设置文件权限（只有当前用户可读取）

#### **问题3：缓存清理策略简单**
- **原因**：超过1000条时全清
- **影响**：可能误删有用的缓存
- **解决方案**：
  1. LRU缓存策略（淘汰最久未使用的）
  2. 按会话ID分组清理

### 6.2 后续优化建议

#### **建议1：使用更快的嵌入模型**
- **方案**：切换到`bge-small-zh`（嵌入维度512，速度更快）
- **效果**：首次查询响应时间从2-5秒降至0.5-1秒
- **工作量**：小（只需要修改`BGE_MODEL`常量）

#### **建议2：批量计算嵌入向量**
- **方案**：一次API调用计算多个文本的嵌入向量
- **效果**：存储多条对话轮次时，速度提升3-5倍
- **工作量**：中（需要修改Ollama API调用方式）

#### **建议3：增加缓存预热功能**
- **方案**：系统启动时，预加载常用会话的嵌入向量
- **效果**：首次查询也能从缓存读取
- **工作量**：中（需要修改系统启动逻辑）
